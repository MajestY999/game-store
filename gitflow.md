# GitFlow для Smoletik-shop

В проекте используется стратегия ветвления, основанная на GitFlow с упрощениями для небольшой команды разработки. Она позволяет поддерживать стабильность кода, удобно разрабатывать новые функции и быстро исправлять ошибки в интернет-магазине.

## Основные ветки

### main — стабильная версия магазина

- Содержит production-код, готовый к деплою
- Все изменения попадают в эту ветку только через Pull Request
- Каждый коммит в main — это либо новый релиз, либо критичный хотфикс
- Тег вида v1.2.3 добавляется при каждом релизе

### develop — основная ветка разработки

- Содержит последние изменения, которые прошли ревью
- Готова к тестированию и интеграции
- Нестабильная ветка, может содержать незавершённые задачи
- Служит базой для создания фича-веток

## Вспомогательные ветки

### feature/\* — ветки для разработки новой функциональности

Создаются от `develop` и после завершения вливаются обратно через Pull Request.

**Примеры для Smoletik-shop:**

- `feature/product-filtering` — добавить фильтрацию по категориям и платформам
- `feature/cart-optimization` — оптимизация корзины покупок
- `feature/payment-integration` — интеграция Stripe платежей
- `feature/user-authentication` — система регистрации и входа
- `feature/recommendation-engine` — система рекомендаций товаров
- `feature/mobile-responsive` — адаптация под мобильные устройства
- `feature/email-notifications` — уведомления по email
- `feature/user-reviews` — система отзывов и рейтинга
- `feature/wishlist` — список желаемых товаров
- `feature/promo-codes` — система промокодов и скидок

### bugfix/\* — ветки для исправления ошибок в разработке

Создаются от `develop` для исправления ошибок, найденных во время разработки или QA.

**Примеры для Smoletik-shop:**

- `bugfix/cart-calculation-error` — исправить неправильный расчет цены в корзине
- `bugfix/search-filtering-issue` — исправить проблему с поиском товаров
- `bugfix/payment-validation` — исправить валидацию формы платежа
- `bugfix/mobile-layout-bug` — исправить верстку на мобильных
- `bugfix/localStorage-sync` — исправить синхронизацию с localStorage
- `bugfix/product-images-load` — исправить загрузку изображений товаров

### hotfix/\* — ветки для срочного исправления ошибок в production

Создаются от `main` для срочного исправления критичных ошибок, которые влияют на работу магазина и продажи.

После исправления вливаются как в `main`, так и в `develop`.

**Примеры для Smoletik-shop:**

- `hotfix/payment-broken` — платежи перестали работать
- `hotfix/stock-sync-down` — товары исчезли из каталога
- `hotfix/security-vulnerability` — найдена уязвимость в безопасности
- `hotfix/catalog-crash` — каталог полностью не загружается

## Правила именования веток

- **Используется нижний регистр** — все буквы строчные
- **Слова разделяются дефисами** — не используются подчеркивания
- **Название кратко описывает задачу** — максимум 3-4 слова
- **Избегайте общих названий** — вместо `feature/fix` используйте `feature/checkout-form-fix`

**Корректные примеры:**

- ✅ `feature/cart-optimizaton`
- ✅ `bugfix/search-filtering-issue`
- ✅ `hotfix/payment-security`

**Некорректные примеры:**

- ❌ `feature/Новая_феча` — кириллица и подчеркивание
- ❌ `Feature/CartOptimization` — заглавные буквы
- ❌ `feature/fix` — слишком общее название

## Процесс разработки

### 1️⃣ Создание задачи

Создайте задачу в GitHub Issues с описанием:

- Что нужно сделать
- Почему это важно
- Критерии приемки

### 2️⃣ Создание ветки

```bash
git checkout develop
git pull origin develop
git checkout -b feature/название-функции
```

### 3️⃣ Разработка и коммиты

```bash
# Разработайте функцию
git add .
git commit -m "feat: описание изменений"

# Еще коммиты при необходимости
git commit -m "fix: исправлена проблема X"
```

### 4️⃣ Пушинг в репозиторий

```bash
git push origin feature/название-функции
```

### 5️⃣ Pull Request

- Откройте PR в ветку `develop`
- Напишите описание (что реализовано, какие задачи решены)
- Запросите код-ревью у коллег
- Добавьте тесты, если применимо

### 6️⃣ Код-ревью и мёрж

- Решите комментарии ревьюеров
- Убедитесь, что все тесты проходят
- Нажмите "Merge Pull Request"
- Удалите ветку после мёржа

### 7️⃣ Релиз в production

Когда накопилось достаточно функций:

```bash
git checkout main
git pull origin main
git merge develop --no-ff
git tag -a v1.1.0 -m "Release v1.1.0: new features"
git push origin main --tags
```

## Коммиты

Используется стиль **Conventional Commits** для логичной истории изменений.

### Основные префиксы

| Префикс       | Описание                                  | Пример                                  |
| ------------- | ----------------------------------------- | --------------------------------------- |
| **feat:**     | Новая функциональность                    | `feat: добавить фильтрацию по жанрам`   |
| **fix:**      | Исправление ошибки                        | `fix: исправить расчет цены в корзине`  |
| **docs:**     | Изменение документации                    | `docs: обновить README с инструкциями`  |
| **style:**    | Изменение стиля (CSS, форматирование)     | `style: отформатировать CSS в Header`   |
| **refactor:** | Улучшение кода без изменения функции      | `refactor: переписать логику поиска`    |
| **perf:**     | Оптимизация производительности            | `perf: оптимизировать загрузку товаров` |
| **test:**     | Добавление или изменение тестов           | `test: добавить тесты для корзины`      |
| **chore:**    | Служебные изменения (зависимости, конфиг) | `chore: обновить зависимости npm`       |

### Примеры для Smoletik-shop

```
feat: добавить систему фильтрации товаров
feat: интегрировать Stripe платежи
feat: добавить рекомендации похожих товаров

fix: исправить ошибку при добавлении в корзину
fix: убрать дублирование товаров в каталоге
fix: исправить валидацию номера карты

docs: обновить README с инструкциями по запуску
docs: добавить комментарии в компоненты

refactor: оптимизировать React компоненты
refactor: переписать логику корзины
refactor: улучшить стилизацию каталога

perf: оптимизировать загрузку изображений
perf: уменьшить размер бандла JavaScript

test: добавить тесты для компонента CartPage
test: добавить интеграционные тесты платежей

chore: обновить React до v20
chore: добавить ESLint конфигурацию
```

### Рекомендации при написании коммитов

- ✅ **Пишите в повелительном наклонении:** `добавить фильтр», не `добавил фильтр`
- ✅ **Короткие, но описательные:** максимум 50 символов в первой строке
- ✅ **Выбирайте подходящий префикс:** не смешивайте feat/fix/refactor
- ✅ **Один коммит = одна логическая единица:** не кидайте 20 разных изменений в один коммит

## Версионирование

Для управления версиями используется **Semantic Versioning (SemVer)**.

### Формат версии: `vMAJOR.MINOR.PATCH`

- **MAJOR** — несовместимое изменение API (например, удаление компонента, изменение структуры данных)
- **MINOR** — новая функциональность, обратно совместимая
- **PATCH** — исправление ошибок, обратно совместимое

### История версий Smoletik-shop

```

v1.0.0 — Первый production релиз
         - Все основные функции готовы

         - Безопасность протестирована

v1.1.0 — Добавлены новые функции
         - Система реферралов
         - Программа лояльности
         - Email уведомления
```

### Создание тага для релиза

```bash
# Создать аннотированный тег
git tag -a v1.2.0 -m "Release v1.2.0: new features and bug fixes"

# Показать информацию о теге
git show v1.2.0

# Запушить тег на сервер
git push origin v1.2.0

# Или запушить все теги
git push origin --tags
```

## Шпаргалка по командам

### Запуск новой фичи

```bash
git checkout develop
git pull origin develop
git checkout -b feature/название
```

### Создание хотфикса для prod

```bash
git checkout main

git checkout -b hotfix/название
# ... исправления ...
git push origin hotfix/название
```

```bash
git branch -a
```

### Удаление локальной ветки

```bash
git branch -d feature/название
```

### Удаление удаленной ветки

```bash
git push origin --delete feature/название
```

### Просмотр истории коммитов

```bash
git log --oneline
git log --graph --all --decorate
```

## Политика PR (Pull Requests)

### Требования перед мёржем

- ✅ Минимум 1 одобрение от другого разработчика
- ✅ Все тесты должны проходить (CI/CD)
- ✅ Нет конфликтов с целевой веткой
- ✅ Код соответствует стилю проекта
- ✅ Документация обновлена (если нужна)

### Описание PR

```markdown
## Описание

Кратко опишите, что реализовано и почему.

## Тип изменения

- [ ] Новая функциональность
- [ ] Исправление ошибки
- [ ] Улучшение документации
- [ ] Рефакторинг

## Связанные задачи

Closes #123

## Скриншоты (если применимо)

[Добавьте скриншоты UI изменений]

## Чеклист

- [ ] Тесты написаны и проходят
- [ ] Документация обновлена
- [ ] Нет warning'ов в консоли
- [ ] Протестировано на мобильных
```

## Частые вопросы

### Как слить две ветки без merge commits?

```bash
git rebase develop
# Вместо merge используйте rebase для "чистой" истории
```

### Что делать, если нужно отменить коммит?

```bash
# Отменить последний коммит (сохранить изменения)
git revert HEAD

# Или переписать историю (опасно!)
git reset --soft HEAD~1
```

### Как синхронизировать вашу ветку с develop?

```bash
git fetch origin
git merge origin/develop
# или
git rebase origin/develop
```
